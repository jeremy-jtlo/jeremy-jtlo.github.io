AWS Database Simulator

Demonstrating DB synchronization with various Amazon Web Services. View it on GitHub →

← Back to Projects

problem

The final project for CMPT 474, a course centering on cloud computing and Amazon Web Services. Using a combination of knowledge from lessons on queueing, database management, and error handling, we were to synchronize the actions performed by 3 database processes amidst dropped, delayed, or out-of-order messages amongst them. Full assignment description is here.

my solution

This one was a team effort. In a group of 4 members including myself, we divided up the work and went at it. Our group needed someone to write the launcher script, someone to handle database actions, someone to cover communication between database instances, and someone to handle the web servers that would send or receive user messages via the bottle.py web server.

approach

When dealing with a 4th year course at SFU, you definitely don't want to mess around. Thankfully, the team I assembled was composed of close allies I knew I could trust from the beginning: my two friends Geoff(god1) and Kevin, as well as my own brother Aaron(ginzoya). Conveniently, we had 4 pairs of hands and 4 major tasks to focus on. Aaron volunteered to handle database logic, Kevin would write out functions to enqueue and dequeue messages from SQS, Geoff agreed to start working on the communication between DBs, and I handled the front/back ends with bottle.py.

execution

The technologies involved with this project were fairly simple. We were given code for tools called Zookeeper and "Publish/Subscribe," which would allow our databases to communicate with each other. In addition to this, request clients and queues were also provided, but the functionality for orchestrating the whole operation was up to us.

As work progressed, we noticed some tasks were easier to complete than others. Particularly, the work I was assigned on the web front end and back end did not take long at all -- it ended up being a couple calls to push or pull JSON with Amazon's Simple Queueing Service. With my hands free of that, I could shift my focus to assisting the other members on their issues, whether it was advice and discussion on IM, or completing merge requests.

Thanks to our coordinated efforts, the project was delivered on time. We may have strayed from our original duties, but it was thanks to the shifting of hands that anything was completed at all.

reflection

Things certainly didn't stick to the original plan very well, but the plan ended up being less significant in the long run. What mattered during this project was how we as a team reacted to rising issues, which I feel was handled very well. The moment any problem arose, someone would report it to the team's group chat so it could be dealt with. Of course, there were more tools provided by our repo for coordinating on problems.

Unfortunately in the migration from SFU's GitLab to Github, all of the issues and discussion threads we had made were lost. The features provided by issues, tagging, and assignment, however, proved to be invaluable to our problem-solving. Some issues persisted over the course of a night and we'd all fall asleep before it could be resolved. This wasn't a problem, though, because there would always be an issue with comments and an assignee to review in the morning.

Overall, this exercise was a good refresher on how to use the provided tools to collaborate on complicated projects. Though a task may be large, tools like GitLab and GitHub allow us to break things down into more manageable, trackable pieces.
